# Red-Teaming Architecture Decision Log

This document records all architectural decisions made during the design and
implementation of the red-teaming subsystem in Scenario.

---

## Decision 1: Composition over Inheritance

**Date:** 2026-02-23
**Decision:** `RedTeamAgent` wraps an inner `UserSimulatorAgent` via composition
rather than subclassing it.

**Rationale:** `UserSimulatorAgent` handles litellm calls, role reversal, config
resolution, and caching. By composing, we reuse all of that without coupling to
its internals. The RedTeamAgent only controls the `system_prompt` field — everything
else is delegated.

**Trade-off:** We depend on being able to mutate `_inner.system_prompt` before
each call. This is fine because the inner agent is fully owned and never used
independently.

---

## Decision 2: Metaprompt Generated Once, Cached

**Date:** 2026-02-23
**Decision:** The attack plan is generated by the metaprompt model once (on the
first `call()`) and cached for all subsequent turns.

**Rationale:** Generating per-phase or per-turn would be 4-50x more expensive
and slower. The plan provides strategic direction; the per-turn phase prompts
handle tactical adaptation. The attacker model can still adapt based on
conversation history even with a static plan.

**Alternative considered:** Per-phase regeneration (regenerate at each phase
boundary). Rejected for cost/complexity — can revisit if needed.

---

## Decision 3: Two-Model Architecture

**Date:** 2026-02-23
**Decision:** `metaprompt_model` generates the attack plan, `attacker_model`
generates the actual messages. `metaprompt_model` defaults to `attacker_model`
if not specified.

**Rationale:** Allows using a more capable (expensive) model for strategic
planning and a faster/cheaper model for turn-by-turn message generation. Most
users will use one model for both by only specifying `attacker_model`.

---

## Decision 4: Crescendo (Marathon) as First Strategy

**Date:** 2026-02-23
**Decision:** Implement only the Crescendo strategy initially. Direct and Roleplay
strategies are scaffolded in the architecture but not implemented.

**Rationale:** Crescendo is the most sophisticated strategy and exercises the full
architecture (phases, metaprompt, turn-awareness). Adding Direct and Roleplay
later requires only implementing `build_system_prompt()` — no core changes.

**Phase structure:**
- Warmup (0-20%): Rapport-building, capability mapping
- Probing (20-45%): Hypothetical/academic boundary testing
- Escalation (45-75%): Authority, roleplay, social engineering
- Direct (75-100%): Maximum pressure, multi-vector attacks

**Why phases work better than brute force:** LLMs maintain conversational
consistency. A cooperative context window built during warmup makes the agent more
likely to comply with later boundary-pushing requests.

---

## Decision 5: Prompt Management — Set Fresh, No Restore

**Date:** 2026-02-23
**Decision:** Each `call()` computes a new system prompt and sets it on the inner
agent. No try/finally restoration.

**Rationale:** The inner `UserSimulatorAgent` is fully owned by `RedTeamAgent`.
Its `system_prompt` is always overwritten before each call. There is no meaningful
"original" to restore to, and no other code reads it between turns.

---

## Decision 6: marathon_script() Helper

**Date:** 2026-02-23
**Decision:** Provide `RedTeamAgent.marathon_script(turns, checks, final_checks)`
as a static method that generates the repetitive script list.

**Rationale:** The existing test pattern requires manually writing
`[scenario.user(), scenario.agent(), check_X] * N` which is 500+ lines for a
50-turn test. The helper generates this programmatically.

`checks` run after every agent response. `final_checks` run once at the end.
`scenario.judge()` is auto-appended.

---

## Decision 7: Description Keeps Attack Hints

**Date:** 2026-02-23
**Decision:** The `description` field in `scenario.run()` still contains attack
hints and context. The `target` parameter on `RedTeamAgent` captures the core
objective.

**Rationale:** Users want control over attack style. The description provides
agent-specific context and technique hints. The metaprompt model uses both
`target` and `description` to generate a tailored plan.

---

## Decision 8: Overridable Metaprompt Template

**Date:** 2026-02-23
**Decision:** The metaprompt template (instructions that tell the metaprompt model
HOW to generate an attack plan) has a hardcoded default but can be overridden via
`metaprompt_template` parameter.

**Rationale:** The default template works well for most cases. Power users may
want to customize it for specialized attack scenarios or to experiment with
different planning approaches.

---

## Decision 9: Attack Plan Internal Only

**Date:** 2026-02-23
**Decision:** The generated attack plan is stored internally and not logged or
exposed via public API.

**Rationale:** Keep the initial implementation simple. Can add logging or an
`attack_plan` property later if debugging/reporting needs arise.
